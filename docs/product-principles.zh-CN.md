# Chief-of-Staff 产品原则

本文档定稿 **v0.1** 级产品叙事：用户面对什么、如何入门、已知交互边界。与 [docs/product-direction.md](product-direction.md)（英文产品方向）互补。

---

## 一句话

Chief-of-Staff 的目标**不是**让用户管理更多 AI 工具，而是让用户始终和一个**清醒的主参谋**保持同步。

---

## 核心亮点

### 1. 主 AI 与用户持续同步

- 用户应该**随时**知道当前在做什么。
- 用户应该知道**下一步**是什么。
- 用户应该知道**哪个任务**在哪个工兵 / Agent 框里。
- 用户**不应该**被迫记住所有上下文。
- 主参谋负责把复杂状态**压缩成清楚的下一步**（例如通过 `chief_next_action`、工兵看板与任务状态文件）。

### 2. 用户只面对产品思路，不面对工具堆

- `chief_doctor`、`chief_audit`、`chief_next_action`、`prepare_cursor_agent_task`、`dispatch_worker` 等是**内部能力**。
- 对用户来说，体验应该是：**「我问主参谋，主参谋告诉我下一步」**。
- 工具越多，越要藏在**清楚的产品流程**背后；亮点不是「工具很多」，而是**状态始终可读、可跟进**。

### 3. 主对话保持短，减少幻觉

- **不**把所有实现细节长期堆在主对话里。
- 把任务**拆小**，用 `.chief/tasks.json` 等状态承载细节。
- **不需要用户长期参与**的执行型任务交给工兵。
- 工兵只把**结果摘要**回传给主参谋（或经用户确认后回传）。
- 主参谋用**状态文件 + 摘要**维持清醒，而不是依赖超长聊天历史。

### 4. 不需要上下文的任务交给工兵

- 文档整理、局部代码修改、测试补充、审计类核对、重复性任务等，适合交给**工兵**在独立 Agent 窗口或外部 API 路线执行。
- 主参谋侧保留：**任务目标、依赖关系、状态、结果摘要**。
- 这样可**节省主对话 token**，也降低主 AI **混乱与幻觉**概率。

### 5. 主参谋 / 工兵边界（v0.1.1 默认 rule 对齐）

- **主参谋（Chief）**负责理解目标、追问关键决策、维护任务账本、建议下一步、准备任务包、回收工兵结果、做一致性审计。
- **默认**主参谋**不**在主对话里直接创建/大改应用代码；**实现**应走 **Cursor Agent Worker**（用户另开 Agent 窗口执行粘贴的任务包）或 **External API Worker**（用户自备 API/模型配置）。
- 仅当用户**明确要求**主参谋**直接改一小处**时，才可在主对话中动文件，且应先说明将改哪些路径。
- **模糊产品句**（如「做个学习软件」）应先走**决策检查点**（平台、MVP、数据、登录、里程碑、工兵路线等），**不要**默认替用户选定并直接开工写代码。
- 新对话开场顺序应先讲清：主参谋角色与两条工兵路线（且默认不直写代码）→ 再问关键选择 → 再给建议，避免让用户误以为参谋已先行定方案。
- **简单实现**可以只派**一个**工兵任务；复杂工作再拆多任务与依赖，避免无谓多 Agent。
- **项目记忆**：勿擅自覆盖既有任务与产品结论；与用户新诉求冲突时先确认。
- **确认语义**：短回复（如「好」「可以」「继续」「嗯」）默认只表示继续对话，不自动等于批准登记任务、准备任务包、派发工兵或改文件。

### 6. 模型策略：强模型协调，工兵执行（常见用法）

- **常见模式**：主参谋对话使用**更强的推理模型**，工兵使用**更快、更便宜或更专门**的模型，把「高质量决策与协调」和「执行任务」分开。
- **Cursor**：主参谋与 Cursor 工兵各自在对应 **Agent 窗口**里由用户选择模型；Chief-of-Staff **不**声称能**自动**替用户切换 Cursor 模型。
- **External**：外部工兵模型由用户在 **provider/model/API** 配置中选择。

## 使用模式

### 模式一：单主参谋模式

**适合**：

- 新手
- 小项目
- 不想在主对话里堆实现细节
- 希望**主对话始终面对参谋**、执行在工兵侧完成的用户

**特点**：

- 用户**主要和主参谋**沟通；工兵在**独立 Cursor Agent 窗口**或 **External** 路线执行（见上「主参谋 / 工兵边界」）。
- 主参谋调用 `chief_doctor`、`chief_audit`、`chief_next_action` 等保持状态清楚；用 `prepare_cursor_agent_task` / `dispatch_worker` 等走工兵。
- **不是**「只有一个聊天所以参谋默认帮你写全部代码」；**v0.1.1** 起 `init` 生成的 rule 会强化这一点。
- 适合**刚开始**使用 Chief-of-Staff。

### 模式二：主参谋 + 工兵模式

**适合**：

- 项目变大、任务变多
- 需要**节省主对话上下文**
- 有些任务**不需要用户实时参与**
- 需要**并行或半并行**推进

**特点**：

- 主参谋**拆任务**、守门（依赖、文件范围）。
- **工兵**处理局部任务；完成后**回传结果摘要**。
- 用户仍**主要面对主参谋**；工兵是「执行面」，不是第二套产品界面。
- 主参谋用 `get_worker_board`、`chief_next_action`、`chief_audit` 等维持**全局同步**。

## 已知交互问题

### 任务包展示 / 压缩问题

- 在 Cursor / MCP 展示链路中，任务包**代码块**或**复制框**可能出现**压缩、折叠**或**渲染不稳定**，影响一键复制与阅读。
- **当前 fallback**：依赖任务包落盘文件、工兵看板、明确的 `task_id` 与描述，必要时从文件系统打开任务包而非仅依赖聊天渲染。
- **后续**：需要专门优化 Cursor handoff 的展示与复制体验（本问题**尚未**在产品层「解决」，仅**记录**为已知事项）。

### Agent 对话框命名问题

- **当前**系统**不能**自动 Rename Cursor Agent；只能通过 `window_hint` 等提示用户**手动**右键 Rename。
- **后续**需要更强的命名规范，让主参谋和用户都能快速对齐「**哪个框对应哪个 task**」。
- 建议命名包含 **`task_id`、`lane`、简短任务名**（具体格式可在实现与文档中逐步统一）。

### 项目级 MCP 与 Agent 上下文对齐

- 用户与主参谋需要**共同确认**：当前对话所属的 Agent **属于哪一个项目**（是否已用 Cursor 打开对应文件夹、是否在该项目窗口内开的聊天）。
- **项目级 MCP**（`.cursor/mcp.json`）与**该项目窗口内的 Agent** 绑定；在正确的项目里才能稳定调用 Chief-of-Staff 工具。
- **全局 Agent**（例如从 Cursor Home 进入、无明确项目根上下文时）**可能**无法加载或无法使用**项目级** MCP，容易与「安装失败」混淆。
- 这也是后续产品上要优化「**该去哪个框、做什么任务**」的原因之一：命名与**项目上下文**需要对齐，用户才不会在错误的对话里找工具。

## 产品边界

- **v0.1** 优先交付可安装的 **Cursor MCP** 产品（见 `chief-mcp-server`）；不宣称未测试平台的支持。
- **Cursor SDK** 自动派发 Cursor 工兵的路线仍处于**设计 / 调研**阶段（见 [cursor-sdk-dispatch-worker-design.zh-CN.md](cursor-sdk-dispatch-worker-design.zh-CN.md)）；**未集成**、**未**作为默认能力发布。
- **外部 API 工兵**路线**保留**：用户自定义 provider / model，与 Cursor 本地工兵互补。
- **用户自己的 API 路线**（external）与 **Cursor 工兵路线**（任务包 handoff）是**两条不同路线**；配置、密钥与费用模型不同，需在对话中区分清楚。
- **不承诺** Claude Code、Trae、Workbody 等未实测平台的支持；以仓库文档与验收状态为准。
